**TEST-02 (SIGUSR1 refresh):**

This test must be `#[cfg(unix)]` since SIGUSR1 is Unix-only.

Strategy: We need the binary to actually start and enter its TUI loop. For that, it needs a non-empty diff. Steps:
1. Create a tempdir, `git init`, create a file, `git add .`, `git commit -m "init"`, then modify the file (so `git diff HEAD` shows changes).
2. Build the binary path using `env!("CARGO_BIN_EXE_semantic-diff")`.
3. Spawn the binary with `.current_dir(temp_dir)` and `stdin(Stdio::piped())` and `stdout(Stdio::piped())` and `stderr(Stdio::piped())`. The TUI needs a terminal, but since we're piping, it may fail to init ratatui. To work around this, we can check that the binary starts without panicking even if the terminal init fails.

ALTERNATIVE APPROACH (more reliable): Instead of trying to run the full TUI (which needs a real terminal), test the signal handling at the process level:
1. Spawn the binary in the temp git repo.
2. The binary will either enter the TUI loop (if terminal is available) or fail on `ratatui::init()`. Since integration tests don't have a TTY, the binary will likely error on terminal init.
3. Instead, test the signal by: spawning the binary, immediately sending SIGUSR1 (the binary writes PID file before terminal init), and verifying the process doesn't crash with a signal-related panic.

SIMPLEST RELIABLE APPROACH: Since the binary calls `signal::write_pid_file()` before `ratatui::init()`, and `ratatui::init()` will fail without a TTY, the binary will exit with an error. But we can still verify SIGUSR1 doesn't cause a crash:
1. Create temp git repo with changes.
2. Spawn binary.
3. Sleep 100ms for PID file write.
4. Send SIGUSR1 via `unsafe { libc::kill(child.id() as i32, libc::SIGUSR1) }`.
5. Wait for process to exit.
6. Check that stderr does NOT contain "panic" or "signal".
7. The process will exit with an error (terminal init failure), but that's OK -- we're testing that SIGUSR1 doesn't make it worse.

Use `libc` crate for `kill()`. Add `libc` to `[dev-dependencies]` in Cargo.toml if not present.

Actually -- better approach: use `std::process::Command` to send the signal via the `kill` system command: `Command::new("kill").args(["-USR1", &pid.to_string()])`. This avoids needing the `libc` crate.

```rust
fn generate_large_diff(num_files: usize) -> String {
    let mut diff = String::new();
    for i in 0..num_files {
        diff.push_str(&format!("diff --git a/src/file_{i}.rs b/src/file_{i}.rs\n"));
        diff.push_str(&format!("--- a/src/file_{i}.rs\n"));
        diff.push_str(&format!("+++ b/src/file_{i}.rs\n"));
        diff.push_str("@@ -1,3 +1,4 @@\n");
        diff.push_str(" fn main() {\n");
        diff.push_str("-    old();\n");
        diff.push_str("+    new();\n");
        diff.push_str("+    added();\n");
        diff.push_str(" }\n");
    }
    diff
}
```
